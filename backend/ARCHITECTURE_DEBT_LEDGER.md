# 架构债台账（可追踪）

目标：将大服务拆分、异常类型治理、可观测标准化按迭代持续消化。
最后更新：2026-03-31

## 0. 本轮已完成的非破坏式加固（2026-04-01）

1. AI诊断入口鉴权收口：角色以登录上下文为准，并拒绝非 customer/technician/admin 角色。
2. 物料库存消耗并发一致性：`MaterialService.consumeMaterialStock` 改为对物料行 `PESSIMISTIC_WRITE` 锁读取，避免并发超扣。
3. 只读查询事务语义：Material 查询类方法标注 `@Transactional(readOnly = true)`，减少误写风险。
4. 对应测试护栏：新增/更新控制器与服务层测试，确保鉴权与低库存闭环逻辑可回归验证。

## 4. 迭代执行规则

1. 每个迭代最多处理 1~2 项架构债，避免大改动集中。
2. 每项债务必须包含：设计说明、回归测试、回滚策略。
3. 默认不改对外 API 契约与数据库结构；如需变更必须单独评审。
4. 每次完成后更新本台账的状态与验收结果。
