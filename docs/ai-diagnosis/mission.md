# AI 维修模块项目使命

## 1. 模块定位

AI 维修模块是车辆维修管理系统中的智能故障初诊与技师辅助决策模块。它同时存在两个使用形态：

1. 顾客端：保持现有大模型提示词初诊模式，面向车主输出通俗、可理解的初步建议。
2. 技师端：接入“规则优先 + 多 Agent 辅助判断”引擎，面向维修人员输出专业、可复核的排查方案。

模块不替代人工维修判断，也不直接驱动核心事务写入。

技师端混合诊断引擎采用以下路线：

1. 规则层优先处理高频、明确、低歧义的基础故障场景。
2. 规则低置信或症状复杂时，触发多 Agent 辅助判断。
3. 多 Agent 输出通过融合层合成为结构化诊断结果。
4. 低置信结果只标记“需人工复核”，不自动修改工单状态。

## 2. 利益相关方需求

### 2.1 车主

车主关注报修便利性、结果可理解性和风险提示明确性。

需求：

- 支持用文字、语音转文字、图片等方式提交故障信息。
- 通过现有大模型提示词初诊快速获得初步故障类型、严重程度、可能原因、费用区间和预计工时。
- 明确知道是否可以继续行驶，是否需要立即检修或拖车。
- 对专业术语给出易懂解释，避免直接输出技师内部语言。
- 诊断结论必须声明为辅助建议，不能让车主误认为系统已经完成真实检测。
- 当前阶段不把规则 + 多 Agent 引擎直接接入顾客端，避免输出过重、过专业或暴露内部证据链。

### 2.2 维修技师

技师关注排查路径、检测数据、风险点和维修经验复用。

需求：

- 基于故障描述生成分步骤排查路径。
- 按低成本、低风险到高成本验证的顺序组织维修建议。
- 输出建议检测工具和数据，例如 OBD 故障码、冻结帧、万用表、电压、电阻、波形、油压、水温等。
- 提醒可能误判点和高危操作前的安全校验。
- 可参考去隐私化的历史维修案例，辅助经验积累。

### 2.3 管理员和门店经营者

管理员关注诊断可控性、成本、调度辅助和合规。

需求：

- 技师端高频故障尽量由本地规则快速处理，降低外部模型调用成本。
- 技师端复杂问题可借助大模型 Agent、库存 Agent、历史案例 Agent 提升判断质量。
- 管理端可在后续查看技师端诊断证据，例如命中规则、Agent 摘要和决策路径。
- 涉及车牌、VIN 等敏感数据时，出站请求必须先脱敏。
- 低置信结果进入人工复核标记，避免错误诊断影响业务安全。

## 3. 产品目标

### 3.1 核心目标

第一阶段目标是保留顾客端现有 AI 初诊体验，同时将技师端诊断链路从“外部 AI 优先，失败后本地规则兜底”重构为“规则优先，多 Agent 辅助判断”。

顾客端目标链路：

```text
输入归一化
-> 现有提示词诊断
-> 大模型输出初步建议
-> 返回通俗化诊断结果
```

技师端目标链路：

```text
输入归一化
-> 隐私脱敏
-> 规则预处理
-> 高置信规则直出
-> 低/中置信触发多 Agent
-> 规则结果与 Agent 证据融合
-> 低置信标记人工复核
-> 返回结构化诊断结果
```

### 3.2 业务目标

- 保持车主报修初筛体验稳定，不在当前阶段引入复杂证据链展示。
- 降低技师端复杂问诊对外部大模型的无差别调用成本。
- 提高复杂故障场景下的诊断完整性。
- 为技师提供可执行、可复核的排查建议。
- 为后续技师复核、库存预警、历史知识检索提供结构化输入。

### 3.3 工程目标

- 保持现有 `/api/ai-diagnosis/diagnose` 接口可用，并根据 role 分流顾客端和技师端逻辑。
- 尽量兼容现有 `AIDiagnosisResponse` 字段。
- 在不引入向量库和复杂异步框架的前提下，先完成可运行 MVP。
- 用测试验证技师端规则直出、隐私脱敏、多 Agent 辅助、低置信标记四条关键路径。

## 4. Demo 范围

MVP Demo 重点展示技师端能力：

1. 技师端输入“刹车变软/刹车异响”等高频明确故障时，规则层直接命中并快速返回。
2. 技师端输入复杂描述时，系统触发多 Agent 辅助判断。
3. 技师端输入中包含车牌或 VIN 时，调用外部模型前完成 mask 脱敏。
4. 库存 Agent 可根据关键词查询现有材料和库存风险。
5. 历史 Agent 可基于关键词检索历史维修工单，返回相似案例摘要。
6. 最终结果包含故障类型、严重等级、可能原因、维修建议、费用区间、工时区间。
7. 综合置信度低于阈值时，响应中标记“需人工复核”，但不改变工单状态。
8. 顾客端仍保持现有初诊流程，输出形式可适当优化，但不展示多 Agent 内部证据链。

## 5. 非目标

当前阶段明确不做以下内容：

- 不训练自有大模型。
- 不建设真实向量数据库或完整 RAG 平台。
- 不实现真正的多 Agent 并发调度。
- 不让 AI 自动创建、关闭、改派或结算工单。
- 不自动扣减库存。
- 不把 AI 诊断结果作为最终维修结论。
- 不在当前阶段将规则 + 多 Agent 引擎直接接入顾客端。
- 不承诺图片内容已经被真实多模态模型理解。当前前端支持图片上传，后端多模态接入需要作为后续阶段验证。

## 6. 当前假设

- `design_spec_extracted.txt` 是 AI 维修模块设计目标的主要依据。
- 当前系统使用 Spring Boot 后端和 Vue 前端。
- 当前 AI 配置沿用 OpenAI-compatible Chat Completions 风格。
- 当前顾客端 AI 诊断保持大模型提示词初诊模式。
- 当前技师端 AI Copilot / AI Suggestion 是规则 + 多 Agent 引擎的优先接入对象。
- 当前可复用实体包括 `Material`、`InventoryAlertNotification`、`RepairOrder`、`Vehicle`、`Technician`。
- 当前 `HybridDiagnosisEngine` 可作为正式服务化改造的基础，但需要接入 Spring 依赖和真实数据 Provider。
- 规则命中阈值、Agent 权重和人工复核阈值可以先写死，后续再参数化。
