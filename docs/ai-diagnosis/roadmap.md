# AI 维修模块实施路线图

## 1. 路线图原则

本路线图采用小步可验证方式推进。每个阶段必须能通过接口调用、单元测试或前端 Demo 明确验证。

优先级排序：

1. 保持现有 AI 诊断接口可用。
2. 顾客端保持现有大模型提示词初诊链路。
3. 技师端先实现规则优先。
4. 技师端再接入多 Agent 辅助判断。
5. 再补充技师端证据链展示和配置化。
6. 不在 MVP 中引入向量库、异步编排或自动工单状态变更。

## 2. 阶段 0：规格冻结

### 阶段目标

明确 AI 维修模块的使命、技术边界和实施路线，避免继续以展示代码替代业务实现。

### 主要任务

- 确认 `design_spec_extracted.txt` 作为 AI 模块目标依据。
- 确认顾客端保持现有大模型提示词初诊链路。
- 确认规则 + 多 Agent 引擎只接入技师端。
- 确认技师端规则高置信允许直出。
- 确认技师端多 Agent 第一版同步顺序执行。
- 确认 History Agent 第一版使用关键词检索历史工单。
- 确认 Inventory Agent 查询现有库存数据。
- 确认低置信只在响应中标记人工复核。

### 产出物

- `docs/ai-diagnosis/mission.md`
- `docs/ai-diagnosis/tech-stack.md`
- `docs/ai-diagnosis/roadmap.md`

### 验收标准

- 三份文档存在。
- 文档中明确当前目标链路。
- 文档中明确非目标。
- 文档中明确当前假设和 MVP 范围。

### 可能风险

- README 与设计说明之间存在差异，后续需要统一口径。
- 当前 `HybridDiagnosisEngine` 仍是普通 Java 类，不能直接注入 Spring 依赖。

## 3. 阶段 1：技师端规则优先接入

### 阶段目标

将技师端诊断链路从“外部 AI 优先，失败后规则兜底”改为“本地规则优先，高置信直出”。顾客端链路不变。

### 主要任务

- 抽取或复用 `HybridDiagnosisEngine.RulePreprocessor`。
- 建立正式服务类，例如 `RuleDiagnosisService`。
- 在 `AIDiagnosisService.diagnoseFault` 中按 role 分流。
- 仅对 `technician` 调用规则优先链路。
- 对 `customer` 保持现有提示词初诊链路。
- 为规则结果生成 `AIDiagnosisResponse`。
- 映射严重等级、费用区间、工时区间。

### 产出物

- 可运行的技师端规则优先诊断链路。
- 高频规则库。
- 规则命中测试。

### 验收标准

- 技师端输入“刹车变软”，不调用外部 AI，返回制动系统相关诊断。
- 技师端输入“发动机抖动故障灯亮”，返回发动机运行异常或动力系统异常。
- 技师端输入“冒烟/焦糊味”，返回高危电气或高温风险。
- 顾客端输入同类问题时，仍走现有大模型提示词初诊链路。
- 返回字段包含 `faultType`、`suggestion`、`severityLevel`、`estimatedCostMin/Max`、`estimatedHoursMin/Max`。

### 可能风险

- 规则过宽导致误命中。
- 规则过窄导致大量请求仍进入外部 AI。
- 规则直出如果缺少解释，会弱化用户信任。

## 4. 阶段 2：隐私脱敏接入

### 阶段目标

在技师端多 Agent 外部 AI 请求前完成敏感数据 mask，避免车牌和 VIN 泄露。顾客端是否接入同一脱敏能力可作为独立安全增强，不属于本阶段的规则 + 多 Agent 改造范围。

### 主要任务

- 建立 `PrivacyMasker` 服务。
- 支持车牌号和 VIN 正则识别。
- 在技师端 Agent 调用 `callOpenAIAPI` 前使用脱敏文本。
- 在 `decisionPath` 中记录“已执行隐私脱敏”。
- 增加测试确认外部 AI 收到的是 mask 后文本。

### 产出物

- 出站请求脱敏能力。
- 脱敏单元测试。

### 验收标准

- 技师端输入包含“沪B67890”时，外部 AI Prompt 不包含原始车牌。
- 技师端输入包含 17 位 VIN 时，外部 AI Prompt 不包含原始 VIN。
- 诊断结果仍能正常返回。

### 可能风险

- 正则误判普通文本。
- 新能源车牌、特殊车牌覆盖不全。
- restore 策略如果处理不当，可能把隐私重新暴露给前端。

## 5. 阶段 3：技师端 Semantic Agent 接入

### 阶段目标

技师端规则低置信或未命中时，调用外部 LLM 做语义分析和复杂故障判断。

### 主要任务

- 将当前 `externalSingleDiagnosis` 改造为 `SemanticAgent`。
- Prompt 改为固定 JSON 输出。
- Prompt 面向技师，输出专业排查证据。
- 将疲劳度、绿色维修提示作为业务上下文注入。
- 解析 Agent 输出为结构化证据。

### 产出物

- `SemanticAgent` 服务。
- JSON Prompt 模板。
- Agent 输出解析测试。

### 验收标准

- 复杂描述能触发外部 AI。
- 外部 AI 输出 JSON 时能稳定解析。
- Markdown 或非标准 JSON 返回时仍有兼容兜底。
- 技师端输出检测工具、检测数据、排查顺序和高危操作提醒。
- 顾客端输出不受本阶段影响。

### 可能风险

- 外部模型不稳定输出非 JSON。
- API Key 缺失或网络失败。
- Prompt 过长导致成本增加。

## 6. 阶段 4：Inventory Agent 接入

### 阶段目标

让技师端诊断结果参考零部件库存和库存预警状态。

### 主要任务

- 建立 `InventoryDiagnosisAgent`。
- 从故障关键词推断可能材料名称。
- 查询 `MaterialRepository.findByNameContaining`。
- 查询现有库存预警。
- 输出库存摘要和风险标签。

### 产出物

- 库存 Agent 服务。
- 库存证据摘要。
- 库存 Agent 测试。

### 验收标准

- 输入“刹车异响”时，可检索刹车片、制动液等相关材料。
- 当相关材料低于安全库存时，Agent 摘要包含库存风险。
- 不自动扣减库存。
- 不自动创建采购任务。

### 可能风险

- 材料命名不统一导致匹配失败。
- 库存数据缺失导致 Agent 长期返回空。
- 关键词匹配可能引入无关材料。

## 7. 阶段 5：History Case Agent 接入

### 阶段目标

利用历史维修工单辅助技师端复杂故障判断。

### 主要任务

- 建立 `HistoryCaseAgent`。
- 基于故障关键词检索历史 `RepairOrder`。
- 关联车辆品牌、车型、年份。
- 输出相似案例摘要。
- 限制返回数量，避免响应过长。

### 产出物

- 历史案例 Agent 服务。
- 简单关键词检索逻辑。
- 历史案例摘要测试。

### 验收标准

- 输入“发动机抖动”时，可以返回历史描述中包含发动机、抖动、故障灯等关键词的工单。
- 返回摘要不包含用户隐私。
- 无相似案例时返回明确空结果摘要。

### 可能风险

- 历史工单描述质量不高。
- 历史数据包含隐私字段，需要过滤。
- 关键词检索召回率有限。

## 8. 阶段 6：决策融合与人工复核标记

### 阶段目标

将技师端规则、语义、库存、历史案例证据融合为最终诊断结果，并对低置信结果打标。

### 主要任务

- 建立正式 `DecisionFusionEngine`。
- 计算综合置信度。
- 生成 `decisionPath`、`ruleHits`、`agentSummaries`。
- 低于 0.85 时设置 `workflowStatus = SUSPENDED`。
- 高于等于 0.85 时设置 `workflowStatus = VALIDATED`。

### 产出物

- 融合服务。
- 响应 DTO 扩展。
- 融合测试。

### 验收标准

- 技师端规则高置信直出时 `workflowStatus = VALIDATED`。
- 技师端复杂低置信场景返回 `workflowStatus = SUSPENDED`。
- 技师端响应包含诊断证据链。
- 低置信不改变任何工单状态。

### 可能风险

- 置信度算法过于粗糙。
- 多 Agent 输出冲突时难以解释。
- 前端暂不展示新增字段，Demo 表达力有限。

## 9. 阶段 7：接口和前端兼容验证

### 阶段目标

确保技师端新链路不破坏现有顾客端、技师端和独立 AI 诊断页。

### 主要任务

- 跑通 `CustomerDashboard` AI 诊断，并确认仍走现有提示词初诊。
- 跑通 `TechnicianDashboard` AI Copilot。
- 跑通 `AIDiagnosisClient`。
- 验证图片-only 输入仍可返回可理解结果。
- 验证旧字段正常展示。

### 产出物

- 接口兼容测试。
- 前端手工验证记录。

### 验收标准

- 三个前端入口均可提交诊断。
- 顾客端输出形式保持通俗初诊，不展示多 Agent 内部证据链。
- 技师端可返回规则命中、Agent 摘要、决策路径等扩展字段。
- 原有结果卡片不报错。
- 图片-only 输入不返回 500。
- 外部 AI 不可用时仍有本地规则结果。

### 可能风险

- 新增字段未展示，答辩时难以体现 Agent 架构。
- 图片实际未进入多模态模型，容易被误解为完整视觉诊断。

## 10. 阶段 8：README 和答辩材料同步

### 阶段目标

统一 README、设计说明和实际实现口径。

### 主要任务

- 在 README 增加 AI 维修模块说明。
- 标明已实现能力和 roadmap 能力。
- 更新 `docs/architecture/agent-architecture.md`，去掉”真实项目运行不依赖该模块”的旧说法，或改为”已按该架构服务化接入”。
- 准备 Demo 脚本。

### 产出物

- README 更新。
- Agent 架构说明更新。
- Demo 操作路径。

### 验收标准

- 文档不再把已接入能力描述成纯展示。
- 文档没有承诺尚未实现的真实 RAG、并发 Agent、多模态视觉识别。
- Demo 脚本能稳定复现规则直出和多 Agent 辅助。

### 可能风险

- 文档宣传超过实现范围。
- 答辩材料中的性能数字需要与真实实现区分。

## 11. 后续增强

后续可以考虑：

- 将规则库迁移到数据库配置。
- 引入向量检索实现真实 RAG。
- 将多 Agent 改为并发执行。
- 增加诊断反馈闭环，让技师标记 AI 诊断是否正确。
- 基于反馈调整规则置信度和关键词权重。
- 将人工复核从响应标记升级为后台任务流。
